# Plan for Autonomous UAV Globe Simulator Implementation

## 1. Objective
Implement the full-scale Autonomous UAV Globe Simulator as outlined in `plan.md`, bridging the C++ simulation engine with a Next.js-based interactive web interface via WebSockets.

## 2. Current Status
- **C++ Engine (`src/`):**
    - Core math (`Geo.h`) and coordinate conversions (`Environment.h`) are partially implemented.
    - Basic `Obstacle`, `Waypoint`, and `ChargingStation` structures exist.
    - `exec.cpp` and `src.cpp` contain experimental/prototype simulation loops and A* routing.
    - WebSocket skeleton exists (`websockets.cpp`).
- **Web App (`web/`):**
    - Next.js project initialized.
    - Needs full implementation of 3D visualization (Globe/UAV/Obstacles) and UI controls.

## 3. Implementation Phases

### Phase 1: Core Engine Refactoring & Robustness
- **Math & Coordinates:** 
    - Refactor `Geo.h` to ensure all `Coordinate` and `Vector3` operations are robust.
    - Solidify `CoordinateConversion` in `Environment.h`.
- **Entity Management:**
    - Implement proper lifecycle management for `Obstacle` subclasses (ensure `updatePosition` for `DynamicObstacle` is functional).
    - Formalize `UAV` class properties: battery, speed, heading, and kinematics.
- **Data Integrity:**
    - Remove any remaining hardcoded data or global variables.

### Phase 2: Environment & Pathfinding Engine
- **Global Environment:**
    - Implement the `GlobalPathfinder` (A*) to work with a node graph of `Waypoints` and `ChargingStations`.
    - Ensure range constraints (battery vs. distance) are correctly factored into pathfinding.
- **Local Environment:**
    - Implement a "Sensor/FOV" mechanism in the `UAV` class.
    - Implement `LocalAvoidance` logic (e.g., D* Lite or basic vector-based avoidance) to react to `Obstacle` objects entering the sensor range.
- **Battery Model:**
    - Implement a realistic battery drain model based on:
        - Distance traveled.
        - Vertical movement (altitude changes).
        - Hovering/Time.

### Phase 3: Communication Layer (WebSocket Server)
- **State Serialization:**
    - Implement a robust JSON serialization for the simulation state (`UAV` position, heading, battery, current path, and active obstacles).
- **Command Interface:**
    - Implement a mechanism to receive commands from the Web UI (e.g., `SET_TARGET`, `SET_WEATHER`, `SPAWN_OBSTACLE`).
- **Server Stability:**
    - Ensure the `MockSimulationServer` in `exec.cpp` is replaced or evolved into a production-ready WebSocket handler.

### Phase 4: Web Frontend (Next.js + 3D)
- **3D Visualization (The "Globe"):**
    - Integrate `react-three-fiber` or `deck.gl`.
    - Render:
        - The Earth (textured sphere).
        - The UAV (icon or simple 3D model).
        - The planned path (polylines).
        - Obstacles (spheres/cubes).
- **Interactive UI:**
    - Dashboard for real-time UAV telemetry (speed, battery, altitude, lat/lon).
    - Control panel: Sliders for obstacle density, wind speed, and simulation speed.
    - Target selector: Ability to click the globe to set a new destination.
- **WebSocket Client:**
    - Implement a hook/service to manage the connection to the C++ backend and sync state with the 3D view.

## 4. Verification Plan
- **Unit Tests:** Verify math functions (`distanceTo`, `bearingTo`, `sphericalToCartesian`) in `src/tests.cpp`.
- **Integration Test:** Run the C++ engine standalone to verify the simulation loop, pathfinding, and battery drain logic in the console.
- **E2E Test:** Launch both C++ engine and Next.js app; verify that UAV movement and obstacle avoidance are reflected accurately on the 3D globe.

## 5. Technical Constraints
- **Language:** C++ (Engine), TypeScript/React (Web).
- **Communication:** WebSockets.
- **Build System:** CMake (for C++).
